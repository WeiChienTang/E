## 一、什麼是 EBC？

**EBC（Enterprise Business Capability，企業業務能力）** 是 Gartner 於 2019 年正式提出的概念，代表企業管理系統的**第四代演進**：

| 世代 | 名稱 | 年代 | 核心思維 |
|------|------|------|----------|
| 第一代 | MRP（物料需求計劃） | 1960-70s | 算出需要多少料 |
| 第二代 | MRPII（製造資源計劃） | 1980s | 整合產能與物料 |
| 第三代 | ERP（企業資源規劃） | 1990-2010s | 整合全企業流程為一套系統 |
| **第四代** | **EBC（企業業務能力）** | **2019+** | **以業務能力為核心，組合式架構** |

### Gartner 的「哥白尼式轉移」

Gartner 用天文學革命來比喻這個轉變：

- **舊思維（地心說）**：ERP 是企業的中心，所有東西繞著 ERP 轉
- **新思維（日心說）**：**業務能力**才是中心，ERP 只是支撐業務能力的其中一個平台

> ERP 不再是宇宙的中心，而是眾多平台之一。

## 二、ERP vs EBC 核心差異

| 維度 | ERP | EBC |
|------|-----|-----|
| **關注點** | 業務流程與應用程式 | 企業的業務能力（做什麼、能做什麼） |
| **架構** | 單體式或套裝式 | 組合式、模組化、能力驅動 |
| **範圍** | 營運效率（財務、人資、供應鏈） | 超越營運效率，涵蓋數位服務、物聯網、客戶體驗 |
| **設計原則** | 「怎麼做」（How） | 「做什麼」（What） |
| **ERP 的角色** | 中央系統 | 眾多支撐平台之一 |
| **整合方式** | 點對點整合 | API 優先、iPaaS 編排 |
| **彈性** | 僵硬、綁定供應商 | 像樂高積木一樣自由組合 |
| **擴展方式** | 買更多模組 | 組合 PBC（Packaged Business Capability） |

### 一句話總結差異

- **ERP 問的是**：「我的東西在哪？花了多少錢？流程跑到哪？」
- **EBC 問的是**：「我的企業具備哪些能力？缺少哪些能力？如何快速組合出新能力？」

## 三、EBC 的五大能力平台

Gartner 與金蝶（Kingdee）共同定義的 EBC 框架包含五大平台：

```
┌─────────────────────────────────────────────────┐
│                   EBC 架構                       │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │ 客戶體驗  │  │ 員工賦能  │  │ 夥伴生態  │      │
│  │ 平台     │  │ 平台     │  │ 平台     │      │
│  │ (CX)     │  │ (EX)     │  │ (PX)     │      │
│  └──────────┘  └──────────┘  └──────────┘      │
│                                                  │
│  ┌──────────┐  ┌──────────────────────┐         │
│  │ 物聯網   │  │ 數據與智慧分析平台    │         │
│  │ 平台     │  │ (Data & AI)          │         │
│  │ (IoT)    │  │                      │         │
│  └──────────┘  └──────────────────────┘         │
│                                                  │
│  ════════════════════════════════════════        │
│  ERP 核心（財務、供應鏈、生產、人資）    ← 底層支撐 │
└─────────────────────────────────────────────────┘
```

### 1. 客戶體驗平台（Customer Experience）
連結與服務客戶 — CRM、全通路互動、客戶自助入口、個性化體驗

### 2. 員工賦能平台（Employee Empowerment）
連結與賦能員工 — 協作工具、知識管理、行動辦公、自助服務

### 3. 夥伴生態平台（Partner Ecosystem）
連結與協同夥伴 — 供應商入口、供應鏈協作、合作夥伴管理

### 4. 物聯網平台（IoT）
連結與管理設備 — 智慧工廠、設備監控、連網產品、預測性維護

### 5. 數據與智慧分析平台（Data & AI）
數據驅動決策 — BI 分析、AI/ML 預測、智慧推薦、大數據洞察

## 五、為什麼 EBC 的意義更深？

### 1. ERP 解決的是「效率」，EBC 解決的是「能力」

ERP 讓企業把現有流程做得更快、更準確。但 EBC 讓企業能夠**快速組合出新的業務能力**來應對市場變化。

> 效率是把事情做對，能力是做對的事情。

### 2. ERP 是封閉的，EBC 是開放的

傳統 ERP 把所有東西鎖在一套系統裡。EBC 的組合式架構讓企業可以：
- 用最好的 CRM 搭配最好的會計系統
- 快速接入新的合作夥伴或銷售通路
- 不必因為換一個模組而動搖整個系統

### 3. ERP 向內看，EBC 向外看

ERP 關注的是企業內部的資源管理。但現代企業的競爭力來自：
- **客戶體驗**（客戶能否自助查詢訂單進度？）
- **供應鏈協作**（供應商能否即時看到需求預測？）
- **數據智慧**（能否預測哪個客戶即將流失？）

這些都是 EBC 的範疇。

### 4. 市場趨勢不可逆

- Gartner 預測：到 2027 年，**75% 的企業**將開始淘汰單體式 ERP，轉向模組化架構
- Forrester 稱下一代企業應用為「數位營運平台（DOP）」，明確指出單一 ERP 套件的模式「行不通」
- SAP、Oracle、Microsoft 都在轉向 EBC 方向（SAP 的 Clean Core + BTP、Microsoft 的 Dynamics 365 生態系）

### 5. 對中小企業特別有利

EBC 的組合式架構讓中小企業可以：
- **不必一次買完整套系統**，按需選擇能力模組
- **降低導入風險**，逐步擴展而非大爆炸式上線
- **用低程式碼平台**彌補 IT 人力不足的問題
- **透過 API 連結生態系**，不必自己全部做

## 六、ERPCore2 的 EBC 演進路線

### 階段一：穩固 ERP 核心（現在 → 持續）

**目標**：確保 ERP 底層資料正確、流程完整

- [x] 進銷存完整流程（採購 → 庫存 → 銷售）
- [x] 會計傳票與財務管理
- [x] 生產排程與 BOM
- [x] 人事薪資
- [x] 審核機制
- [ ] 補強情境檢查覆蓋率
- [ ] 強化跨模組資料一致性驗證

> 這是地基，必須穩固。

### 階段二：可配置化核心（對應 [EBC可配置化設計](系統功能設計/Readme_EBC可配置化設計.md)）

**目標**：讓不同廠商可以自行定義欄位、規則、流程，不需改程式碼

| 可配置化層次 | 解決的問題 | 詳細設計 | 狀態 |
|---|---|---|---|
| Level 1：自訂資料表 | 不同廠商需要記錄不同資料 | [可配置化設計 §三](系統功能設計/Readme_EBC可配置化設計.md) | ✅ 已實作（主表） |
| Level 2：業務規則 | 不同廠商的審核/通知/驗證規則不同 | [可配置化設計 §四](系統功能設計/Readme_EBC可配置化設計.md) | 規劃中 |
| Level 3：流程配置 | 不同廠商的單據流轉順序不同 | [可配置化設計 §五](系統功能設計/Readme_EBC可配置化設計.md) | 規劃中 |
| Level 4：畫面配置 | 不同廠商想隱藏/必填不同欄位 | [可配置化設計 §六](系統功能設計/Readme_EBC可配置化設計.md) | ✅ 已實作 |

**Level 1 已實作內容（2026-03-28）：**
- 自訂資料表系統（EAV 模式）：`CustomTableDefinition` + `CustomFieldDefinition` + `CustomTableRow` + `CustomFieldValue`
- 管理員可透過「系統 → 自訂資料表」建立表定義與欄位設定
- 支援 7 種欄位類型：文字、多行文字、數值、日期、日期時間、布林、下拉選單
- 所有資料列繼承 BaseEntity（自動擁有 Id、Code、Status、審計欄位、草稿支援）
- 權限控制：`System.CustomTable`（Sensitive 等級）

**Level 4 已實作內容：**
- `FieldDisplaySetting` 欄位顯示覆蓋（顯示/隱藏、必填覆蓋、名稱覆蓋）
- `FieldProtectionLevel` 欄位保護等級（SystemRequired / BusinessRequired / Normal）
- `FieldSettingsPanel` 管理介面（嵌入 GenericEditModalComponent）

> 這是 EBC 與傳統 ERP 最關鍵的差異：使用者自己組合能力，而非工程師客製開發。

### 階段三：API 化與模組解耦（PBC 基礎建設）

**目標**：為 EBC 對外平台打下技術基礎

| 任務 | 說明 |
|------|------|
| 服務層 API 化 | 將現有 Service 層開放為 RESTful API |
| 模組邊界定義 | 明確每個模組的輸入、輸出與事件 |
| 事件驅動架構 | 關鍵業務動作發布事件（訂單成立、出貨完成、付款確認） |
| 認證與授權 | 支援 API Key / OAuth 供外部系統存取 |

### 階段四：客戶體驗平台（CX）

**目標**：讓客戶從「被管理的對象」變成「互動的參與者」

| 功能 | 說明 | 價值 |
|------|------|------|
| 客戶入口網站 | 客戶自助查詢訂單、出貨、對帳 | 減少客服人力，提升客戶滿意度 |
| 線上報價/下單 | 客戶直接在線上提交需求 | 縮短訂單週期 |
| 即時通知 | 訂單狀態變更自動推送（Email/LINE） | 透明度提升 |
| 客訴追蹤入口 | 客戶自助提交並追蹤客訴處理進度 | 服務品質可視化 |

### 階段五：夥伴生態平台（PX）

**目標**：讓供應商成為供應鏈的協作夥伴

| 功能 | 說明 | 價值 |
|------|------|------|
| 供應商入口 | 供應商自助查詢採購單、交貨排程 | 降低溝通成本 |
| 供應商自助報價 | 詢價單線上回覆 | 加速採購流程 |
| 交貨預警協作 | 供應商回報交期異動，自動觸發重排程 | 供應鏈韌性 |
| 對帳與付款查詢 | 供應商自助查詢付款狀態 | 減少財務對帳人力 |

### 階段六：數據與智慧分析平台（Data & AI）

**目標**：從「看報表」升級到「預測與決策」

| 功能 | 說明 | 價值 |
|------|------|------|
| 銷售預測 | 基於歷史數據預測未來需求 | 精準備料、降低庫存成本 |
| 客戶流失預警 | 分析客戶下單頻率異常 | 及早介入挽回 |
| 智慧採購建議 | 根據庫存水位與交期自動建議採購 | 避免缺料或過量 |
| 生產排程最佳化 | 考量產能、交期、優先序的智慧排程 | 提升稼動率 |

### 階段七：員工賦能與物聯網（進階）

| 功能 | 說明 |
|------|------|
| 行動端 APP / PWA | 業務人員在外可即時查詢與下單 |
| 協作與知識庫 | 跨部門協作、經驗知識累積 |
| 通用 IoT 框架 | 擴展磅秤整合到更多設備（溫度、濕度、機台狀態） |
| 設備預測性維護 | 基於 IoT 數據預測設備故障時間 |

## 九、結論：關鍵重點歸納

### 重點一：EBC 不是取代 ERP，而是以 ERP 為基礎向上發展
ERP 是 EBC 的底層支撐。沒有穩固的 ERP，EBC 就是空中樓閣。ERPCore2 目前的 ERP 核心是正確且必要的基礎。

### 重點二：EBC 的核心價值是「連結」
ERP 管理資源，EBC 連結人（客戶、員工、夥伴）與物（設備、產品）。企業的價值不再只來自內部效率，更來自**生態系的協作能力**。

### 重點三：組合式架構是 EBC 的技術實現方式
透過 PBC（Packaged Business Capability）將功能模組化，像樂高積木一樣自由組合。這要求系統具備清晰的模組邊界、API 介面、與事件驅動機制。

### 重點四：從「向內管理」轉向「向外服務」
最大的思維轉變是：系統的使用者不再只有內部員工，還包括客戶與供應商。這改變了 UI 設計、安全模型、與系統架構的根本假設。

### 重點五：中小企業更需要 EBC 思維
正因為資源有限，中小企業更需要用組合式的方式按需建構能力，而非被迫購買用不到的完整套件。ERPCore2 作為自主開發的系統，天然具備這個優勢。

### 重點六：市場方向已定，不是要不要做，而是什麼時候做
Gartner 預測 2027 年 75% 企業將淘汰單體式 ERP。SAP、Oracle、Microsoft 已全面轉向。ERPCore2 越早在架構上預留 EBC 的擴展空間，未來的轉型成本越低。
